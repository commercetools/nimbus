## Comparison: pattern vs manual composition

FeedbackCard is a composition of existing primitives. It renders no text and no
button of its own — you provide those as children — and it exposes no `variant`
prop. Set `colorPalette` on `FeedbackCard.Root` and the whole card themes
cohesively: use `colorPalette.*` tokens for the surface, and the copy and
`Button` follow the palette automatically. `Root` sets a palette-aware text
color, so plain `Heading`/`Text` render in the card's palette with no color prop
(see [Theming](#theming)).

**With FeedbackCard:**

```tsx
<FeedbackCard.Root
  colorPalette="positive"
  bg="colorPalette.2"
  borderRadius="200"
  p="400"
>
  <FeedbackCard.Content>
    <Heading size="md">Suggestion approved</Heading>
    <Text>Applied the recommended discount.</Text>
  </FeedbackCard.Content>
  <FeedbackCard.Action>
    <Button variant="outline" onPress={handleUndo}>
      Undo
    </Button>
  </FeedbackCard.Action>
</FeedbackCard.Root>
```

**Manual composition (escape hatch):**

```tsx
<Flex
  colorPalette="positive"
  bg="colorPalette.2"
  borderRadius="200"
  p="400"
  wrap="wrap"
  align="center"
  justify="space-between"
  gap="400"
>
  <Stack gap="100" flex="1 1 auto" minWidth="0">
    <Heading size="md">Suggestion approved</Heading>
    <Text>Applied the recommended discount.</Text>
  </Stack>
  <Button variant="outline" onPress={handleUndo}>
    Undo
  </Button>
</Flex>
```

### When to use which

**Use FeedbackCard when:**

- You want the canonical embedded confirmation layout (responsive wrapping row,
  content on the start, action on the end)
- You want consistent spacing and wrapping behavior across the app
- You want a semantic, self-documenting API (`Root` / `Content` / `Action`)

**Use manual composition when:**

- You need a fundamentally different layout the slots do not express

## Getting started

### Import

```tsx
import { FeedbackCard } from "@commercetools/nimbus";
```

### Basic usage

Compose the three parts. Give `FeedbackCard.Content` a `Heading` title and a
`Text` subtitle, and drop your own `Button` into `FeedbackCard.Action`. Apply
the visual context with style props on `FeedbackCard.Root` — set `colorPalette`
and reach for `colorPalette.*` tokens for the surface; the copy and button take
care of themselves.

```jsx live-dev
const App = () => (
  <FeedbackCard.Root
    colorPalette="positive"
    bg="colorPalette.2"
    borderRadius="200"
    p="400"
  >
    <FeedbackCard.Content>
      <Heading size="md">Suggestion approved</Heading>
      <Text>Applied the recommended discount to 3 products.</Text>
    </FeedbackCard.Content>
    <FeedbackCard.Action>
      <Button variant="outline" onPress={() => console.log("Undo")}>
        Undo
      </Button>
    </FeedbackCard.Action>
  </FeedbackCard.Root>
);
```

## Theming

FeedbackCard derives its colors entirely from the palette you set — it has no
fixed hues of its own. Establish a [color palette](/guidelines/colors) on
`FeedbackCard.Root` with the `colorPalette` prop, then reference
`colorPalette.*` tokens for the surface. Everything else follows automatically:

- **Surface**: `bg="colorPalette.2"`, optional `borderColor="colorPalette.6"`
- **Copy**: nothing — `Root` sets a palette-aware text color
  (`color: "colorPalette.11"`), which the copy inherits, so plain `Heading` and
  `Text` render in the card's palette. No `color` prop needed.
- **Action**: nothing — the `Button` inherits `colorPalette` from `Root` via the
  cascade.

Switch contexts by switching the palette: `colorPalette="positive"` for an
approval, `colorPalette="critical"` for a rejection, `colorPalette="info"` for a
neutral notice, and so on.

## Usage examples

### Approve context

An approval reads as a single positive-tinted unit — surface and button — with
clear, legible copy.

```jsx live-dev
const App = () => (
  <FeedbackCard.Root
    colorPalette="positive"
    bg="colorPalette.2"
    border="solid-25"
    borderColor="colorPalette.6"
    borderRadius="200"
    p="400"
  >
    <FeedbackCard.Content>
      <Heading size="md">Suggestion approved</Heading>
      <Text>Applied the recommended discount.</Text>
    </FeedbackCard.Content>
    <FeedbackCard.Action>
      <Button variant="outline" onPress={() => console.log("Undo")}>
        Undo
      </Button>
    </FeedbackCard.Action>
  </FeedbackCard.Root>

```

### Reject context

```jsx live-dev
const App = () => (
```

### Responsive wrapping

`FeedbackCard.Root` is a `flex-wrap` row. When the content and action no longer
fit side by side, the action wraps below the content. Resize the preview to see
it in action.

```jsx live-dev
const App = () => (
  <Box maxWidth="320px">
    <FeedbackCard.Root
      colorPalette="neutral"
      bg="colorPalette.2"
      borderRadius="200"
      p="400"
    >
      <FeedbackCard.Content>
        <Heading size="md">
          This title and subtitle are intentionally long
        </Heading>
        <Text>
          So the action button wraps to a new line on narrow containers.
        </Text>
      </FeedbackCard.Content>
      <FeedbackCard.Action>
        <Button variant="outline" onPress={() => console.log("Undo")}>
          Undo
        </Button>
      </FeedbackCard.Action>
    </FeedbackCard.Root>
  </Box>
);
```

## Component requirements

### Accessibility

FeedbackCard is layout-only and introduces no interactive elements of its own,
so accessibility rests on the primitives you compose inside it:

- **No implicit role**: `FeedbackCard.Root` renders a plain `<div>` with no ARIA
  role, so it does not falsely announce as an alert or live region. If you want
  the card announced as a single unit, pass `role="group"` together with an
  `aria-label` — both forward onto the root element.
- **The action**: use a Nimbus `Button` (or another React Aria control) inside
  `FeedbackCard.Action`. It supplies its own accessible name, focus behavior,
  and keyboard support.
- **The copy**: compose `FeedbackCard.Content` from semantic primitives — a
  `Heading` for the title and `Text` for the subtitle — so the hierarchy reads
  correctly for assistive technology.

If your use case requires analytics for the action, it is good practice to add a
**persistent**, **unique** id to your button:

```tsx
const PERSISTENT_ID = "agent-suggestion-undo";

export const Example = () => (
  <FeedbackCard.Action>
    <Button id={PERSISTENT_ID} variant="outline" onPress={handleUndo}>
      Undo
    </Button>
  </FeedbackCard.Action>
);
```

#### Keyboard navigation

FeedbackCard adds no keyboard behavior of its own. The consumer-provided action
button participates in the normal tab order:

- `Tab` / `Shift+Tab`: move focus to and from the action button
- `Enter` or `Space`: activate the action button

## Testing your implementation

FeedbackCard's own rendering, layout, and style-prop forwarding are covered by
Nimbus story tests — you don't need to re-test them. These examples focus on the
integration you own: wiring the action into your application's state.

### React to undo with your own state

The pattern positions the button but doesn't change it. Own the state transition
in your component — here, pressing Undo reverts the outcome and swaps the card
for an acknowledgement.

```tsx
import { useState } from "react";
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
import {
  FeedbackCard,
  Heading,
  Text,
  Button,
  NimbusProvider,
} from "@commercetools/nimbus";

it("reverts the applied change when undo is pressed", async () => {
  const user = userEvent.setup();

  const AgentSuggestion = () => {
    const [status, setStatus] = useState("applied");

    if (status === "reverted") {
      return <Text>Change reverted.</Text>;
    }

    return (
      <FeedbackCard.Root
        colorPalette="positive"
        bg="colorPalette.2"
        borderRadius="200"
        p="400"
      >
        <FeedbackCard.Content>
          <Heading size="md">Suggestion approved</Heading>
        </FeedbackCard.Content>
        <FeedbackCard.Action>
          <Button variant="outline" onPress={() => setStatus("reverted")}>
            Undo
          </Button>
        </FeedbackCard.Action>
      </FeedbackCard.Root>
    );
  };

  render(
    <NimbusProvider>
      <AgentSuggestion />
    </NimbusProvider>
  );

  await user.click(screen.getByRole("button", { name: /undo/i }));
  expect(screen.getByText("Change reverted.")).toBeInTheDocument();
});
```

### Dismiss one card from a feed

Render a list of confirmations and remove the matching entry when its undo is
pressed — a common chat-feed workflow.

```tsx
it("removes only the undone entry from the feed", async () => {
  const user = userEvent.setup();

  const SuggestionFeed = () => {
    const [items, setItems] = useState([
      { id: 1, label: "Discount applied" },
      { id: 2, label: "Tax rate updated" },
    ]);

    return (
      <Stack gap="200">
        {items.map((item) => (
          <FeedbackCard.Root
            key={item.id}
            colorPalette="neutral"
            bg="colorPalette.2"
            p="400"
          >
            <FeedbackCard.Content>
              <Text>{item.label}</Text>
            </FeedbackCard.Content>
            <FeedbackCard.Action>
              <Button
                variant="outline"
                aria-label={`Undo ${item.label}`}
                onPress={() =>
                  setItems((prev) => prev.filter((i) => i.id !== item.id))
                }
              >
                Undo
              </Button>
            </FeedbackCard.Action>
          </FeedbackCard.Root>
        ))}
      </Stack>
    );
  };

  render(
    <NimbusProvider>
      <SuggestionFeed />
    </NimbusProvider>
  );

  await user.click(
    screen.getByRole("button", { name: /undo discount applied/i })
  );

  expect(screen.queryByText("Discount applied")).not.toBeInTheDocument();
  expect(screen.getByText("Tax rate updated")).toBeInTheDocument();
});
```
