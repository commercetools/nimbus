## Overview

Feedback card is a pattern that surfaces after an agent suggestion has been
approved or rejected, prompting the user to confirm or undo the action. It acts
as a soft confirmation step within the feed — a low-emphasis, inline
acknowledgement rather than a blocking dialog.

It is a layout-only composition of existing primitives: a responsive row that
places a short message on the leading edge and a single action on the trailing
edge. It has no visual style of its own — the surface color, border, radius, and
padding are chosen per context (for example, a positive treatment for an
approval, a critical treatment for a rejection).

## Anatomy

- **Content** — a short message on the leading edge, typically a title and a
  supporting subtitle describing what happened.
- **Action** — a single button on the trailing edge, usually _Undo_, giving the
  user a way to reverse the outcome.

The row wraps responsively: when the message and action no longer fit side by
side, the action drops below the message rather than crowding it.

```jsx live
const App = () => (
  <FeedbackCard.Root
    colorPalette="positive"
    bg="colorPalette.2"
    border="solid-25"
    borderColor="colorPalette.6"
    borderRadius="200"
    p="400"
  >
    <FeedbackCard.Content>
      <Heading size="sm">Suggestion approved</Heading>
      <Text>Applied the recommended discount to 3 products.</Text>
    </FeedbackCard.Content>
    <FeedbackCard.Action>
      <Button variant="outline" onPress={() => {}}>
        Undo
      </Button>
    </FeedbackCard.Action>
  </FeedbackCard.Root>
);
```

## Variables

Feedback card has no variant or size options. Its visual treatment is applied
per instance, which lets a single pattern express any context — most commonly
approval and rejection — while keeping spacing and layout consistent.

### Approval

Use a positive surface to acknowledge that a suggestion was accepted.

```jsx live
const App = () => (
  <FeedbackCard.Root
    colorPalette="positive"
    bg="colorPalette.2"
    border="solid-25"
    borderColor="colorPalette.6"
    borderRadius="200"
    p="400"
  >
    <FeedbackCard.Content>
      <Heading size="sm">Suggestion approved</Heading>
      <Text>Applied the recommended discount.</Text>
    </FeedbackCard.Content>
    <FeedbackCard.Action>
      <Button variant="outline" onPress={() => {}}>
        Undo
      </Button>
    </FeedbackCard.Action>
  </FeedbackCard.Root>
);
```

### Rejection

Use a critical surface to acknowledge that a suggestion was dismissed. The
layout is identical — only the color treatment changes.

```jsx live
const App = () => (
  <FeedbackCard.Root
    colorPalette="critical"
    bg="colorPalette.2"
    border="solid-25"
    borderColor="colorPalette.6"
    borderRadius="200"
    p="400"
  >
    <FeedbackCard.Content>
      <Heading size="sm">Suggestion rejected</Heading>
      <Text>No changes were applied.</Text>
    </FeedbackCard.Content>
    <FeedbackCard.Action>
      <Button variant="outline" onPress={() => {}}>
        Undo
      </Button>
    </FeedbackCard.Action>
  </FeedbackCard.Root>
);
```

## Guidelines

### When to use

> [!TIP]\
> Use a feedback card to confirm an action inline and offer a single, reversible
> next step.

- After an agent suggestion is approved or rejected in a chat feed.
- When the outcome is reversible and you want to offer an _Undo_ without
  interrupting the flow.
- For low-emphasis, non-blocking acknowledgements that stay in the feed.

### When not to use

> [!CAUTION]\
> Don't use a feedback card for anything that must block the user or demand a
> decision before continuing.

- For destructive or irreversible actions that need explicit confirmation — use
  an `AlertDialog`.
- For transient, auto-dismissing notifications — use a `Toast`.
- For persistent, page-level status messaging unrelated to a single action — use
  an `Alert`.

### Do

- Keep the message to a short title plus an optional one-line subtitle.
- Offer a single action (typically _Undo_); the pattern is a soft confirmation,
  not a full action bar.
- Match the surface color to the outcome (positive for approvals, critical for
  rejections) so the context reads at a glance.

### Don't

- Don't stack multiple competing actions in the action slot — keep it to one.
- Don't rely on color alone to convey the outcome; the message text should state
  what happened.
- Don't use it for content that must be acknowledged before the user can
  proceed.
