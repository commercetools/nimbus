## Getting started

`Markdown` renders a Markdown source string into Nimbus-styled, accessible React
elements. It is a single entry point (not a compound component) plus a default
renderer map, built on the headless `react-markdown` engine with `remark-gfm`
enabled by default.

### Import

```tsx
import { Markdown, type MarkdownProps } from "@commercetools/nimbus";
```

### Anatomy

```tsx
<Markdown
  components={{ a: MyLink, SearchQueryResultCard: Card }}
  isStreaming={false}
  headingOffset={0}
>
  {markdownString}
</Markdown>
```

- **`children`** — the Markdown source string (canonical input).
- **`components`** — renderer map, shallow-merged over the Nimbus defaults.
  Standard element keys override that element; non-standard keys register
  embeddable custom component tags (see below).
- **`allowedElements` / `disallowedElements`** — tune the rendered element set.
- **`isStreaming`** — enables safe incremental rendering of streamed output.
- **`headingOffset`** — shifts rendered heading levels to preserve the page
  outline.
- **Style props** — forwarded to the outer root container (set `maxW`, spacing,
  `lineClamp`, etc.).

## Default renderers

Out of the box every standard element maps to a Nimbus-styled renderer using the
Figma `Markdown/*` typography scale. Headings `#`–`####` map to the heading
scale; `#####`/`######` fold to the smallest. External links automatically get
`target="_blank"`, `rel="noopener noreferrer"`, an "(opens in new tab)" label,
and a visible icon indicator. Images render with `loading="lazy"` and
`referrerpolicy="no-referrer"`.

## Overriding an element

Pass a `components` entry keyed by HTML element name. Overriding one element
leaves every other default intact. Every renderer also receives the original
hast `node` — destructure it out before spreading onto a DOM element.

```tsx
<Markdown
  components={{
    a: ({ children, href }) => <Link href={href}>{children}</Link>,
  }}
>
  {content}
</Markdown>
```

## Custom component tags

Register a non-standard key in `components` and you can embed that tag directly
in the Markdown source. The tag's string attributes are passed to the component
as props, and tag names match by exact case (PascalCase preserved).

```tsx
<Markdown components={{ SearchQueryResultCard: Card }}>
  {`Here is what I found:\n\n<SearchQueryResultCard id="sku-42" />`}
</Markdown>
// → renders <Card id="sku-42" />
```

- **Self-closing** (`<Tag ... />`) and **paired** (`<Tag ...>…</Tag>`, with
  markdown children) are both supported. For a block-level paired tag, keep
  blank lines around the tags so the content between them is parsed as markdown.
- **Safe by default** — a custom tag is materialized only if a component is
  registered for it; unregistered tags stay inert. Like every other renderer,
  custom components receive the hast `node` — destructure it out before
  spreading onto a DOM element.
- Custom components own their own accessibility, exactly like element overrides.

## Safe by default

Raw HTML is never rendered as live markup: it is skipped, rendering is restricted
to a safe element allowlist, and dangerous URLs (`javascript:`, `vbscript:`,
`file:`) are neutralized. Image-host security is the application CSP's
responsibility (`img-src`). To render application components, register them as
custom component tags (above) rather than embedding raw HTML.

## Streaming

```tsx
<Markdown isStreaming={isResponding}>{partialResponse}</Markdown>
```

While streaming, unterminated constructs are completed via `remend`, content is
split into memoized blocks (only the final block re-parses per token), the root
sets `aria-busy`, and a single coalesced completion announcement is made when
the stream settles. Consumers never manage ARIA.
