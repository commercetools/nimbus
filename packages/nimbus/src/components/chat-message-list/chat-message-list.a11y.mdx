## Accessibility

`ChatMessageList` owns the half of chat accessibility that a single
`ChatMessage` intentionally leaves to the transcript: the container's live
region and the announcement of new content. This page covers what the list does
for you and the small amount you are responsible for.

```jsx live
const App = () => (
  <ChatMessageList.Root aria-label="Conversation with the agent" height="260px">
    <ChatMessageList.Item>
      <ChatMessage.Root sender="user" aria-label="Message from Ada Lovelace">
        <ChatMessage.Avatar firstName="Ada" lastName="Lovelace" />
        <ChatMessage.Body>
          <Text>Can you summarise last week's orders?</Text>
        </ChatMessage.Body>
      </ChatMessage.Root>
    </ChatMessageList.Item>
    <ChatMessageList.Item>
      <ChatMessage.Root sender="agent" aria-label="Message from the agent">
        <ChatMessage.Avatar aria-label="Agent">
          <Icons.AutoAwesome />
        </ChatMessage.Avatar>
        <ChatMessage.Body>
          <Text>Revenue is up 12% week-over-week.</Text>
        </ChatMessage.Body>
      </ChatMessage.Root>
    </ChatMessageList.Item>
  </ChatMessageList.Root>
);
```

### Naming the transcript

`ChatMessageList.Root` renders a `role="log"` region that needs an accessible
name. Provide one with `aria-label` or `aria-labelledby`. If you supply neither,
the list falls back to a localized default ("Conversation") so the region is
never anonymous — but a name specific to the conversation is better.

### Why `log`, not `feed`

Two ARIA container roles fit a message transcript, and they are not
interchangeable:

- **`role="log"` with `aria-live="polite"`** (the default here) — a live region
  that announces appended content, including streamed replies, without
  re-announcing the whole history on every token. It has **no constraint on
  child roles**, so it validly contains `<article>` messages, centered notices,
  date separators, or anything else the transcript needs.
- **`role="feed"`** — the ARIA APG's dedicated pattern for scrollable article
  collections, with built-in "read next/previous article" keyboard support. Its
  trade-off: `feed` has an `aria-required-children` constraint — **every** child
  must have role `article`. A centered notice or a divider inside a `feed` is an
  accessibility violation, not just a lint warning.

Because a real transcript mixes messages with notices and dividers,
`ChatMessageList` defaults to `log`. If you are specifically building
article-to-article keyboard navigation, you can compose a `role="feed"` yourself
from message `article`s only — but then you cannot mix in non-article members.

### One announcer during streaming

Streaming has a single announcer. The list's `log` region announces
message-level changes (a message appended, a reply arriving). The streamed
message itself only sets `aria-busy` via `isStreaming` on its
`ChatMessage.Root`; it does **not** create its own live region — a region
mounted at the same moment as its content is not reliably announced, and two
overlapping live regions would double-announce.

How much of the streamed text is announced depends on how you render it.
`aria-busy="true"` asks assistive technology to defer announcing changes inside
the message until the stream settles and the flag clears — which keeps a
compliant screen reader from reading every token — but support for that deferral
varies between screen readers. The recommended path is to render the body with
`<Markdown isStreaming>`: it owns a single, coalesced completion announcement
instead of emitting text token by token. If you stream raw growing text into the
`log` with neither `aria-busy` nor a component that coalesces its own
announcement, expect some screen readers to announce intermediate updates.

### The list is always mounted

The `log` region persists across messages — it is present before content
changes, which is what makes appended and streamed content announce reliably.
Render `ChatMessageList.Root` for the life of the conversation rather than
mounting it alongside the first message.

### `Item` does not assert `listitem`

`ChatMessageList.Item` provides DOM order and a styling hook but deliberately
does not set `role="listitem"`, which would require a `list`/`log` parent
relationship that conflicts with the live-region semantics. Members keep their
own roles (a message `article`, a notice) unchanged.

### Accessibility standards

- **Status messages:** `Root` is `role="log"` `aria-live="polite"` so new and
  streamed messages are announced without the consumer building custom
  live-region logic.
- **Name, role, value:** name the `log` region with `aria-label` /
  `aria-labelledby`; a localized default applies only as a fallback.
- **Info and relationships:** a mixed transcript stays valid because `log`
  imposes no required-children constraint — messages, notices, and dividers can
  coexist.
- **Focus order / no keyboard trap:** the scroll-to-bottom control is a single
  focusable button that appears only while the pin is released; it never traps
  focus.
