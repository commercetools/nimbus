## Getting started

### Import

```tsx
import { Splitter } from "@commercetools/nimbus";
import type { SplitterRootProps } from "@commercetools/nimbus";
```

### Basic usage

```tsx
<Splitter.Root
  orientation="horizontal"
  defaultSizes={{ nav: 30, main: 70 }}
  panes={{
    nav: { minSize: 15 },
    main: { minSize: 25 },
  }}
>
  <Splitter.Pane id="nav">{/* nav content */}</Splitter.Pane>
  <Splitter.Handle />
  <Splitter.Pane id="main">{/* main content */}</Splitter.Pane>
</Splitter.Root>
```

## Design model

### Sizes are uncontrolled; collapse is controllable

Sizes drag at ~60Hz, so threading them through a controlled prop would call the
consumer's `setState` every tick and re-render everything above the splitter.
Sizes are therefore **uncontrolled**: `defaultSizes` (read once on mount) plus
two notification callbacks:

- `onSizesChange(sizes)` — fires on every change, including each drag tick. Use
  it for a live read-out (e.g. a pixel display).
- `onSizesChangeEnd(sizes)` — fires once when an interaction settles (drag end,
  each keypress, collapse/expand, double-click restore). **This is the seam to
  wire persistence to** — no debouncing required.

Collapse, by contrast, is discrete and low-frequency, so it follows the normal
Nimbus controlled/uncontrolled convention: `collapsedPane` /
`defaultCollapsedPane` / `onCollapsedPaneChange`. A 2-pane splitter collapses at
most one pane at a time, so the value is a single id or `null`.

### Id-keyed configuration

Each `Splitter.Pane` carries only its `id`. Per-pane constraints (`minSize`,
`collapsible`, `collapsedSize`) live on `Splitter.Root` in the `panes` map keyed
by that id:

```tsx
<Splitter.Root
  panes={{
    nav: { minSize: 10, collapsible: true },
    main: { minSize: 20 },
  }}
  defaultSizes={{ nav: 30, main: 70 }}
  onSizesChangeEnd={(sizes) => console.log(sizes)}
>
  <Splitter.Pane id="nav">…</Splitter.Pane>
  <Splitter.Handle />
  <Splitter.Pane id="main">…</Splitter.Pane>
</Splitter.Root>
```

Id-keyed instead of index-based for four reasons:

1. **Reorder-safe persistence.** Swapping the panes between releases doesn't
   invalidate stored layouts — sizes are keyed by id, not position.
2. **Readable APIs.** `collapsedPane="nav"` beats `collapsedPane={0}`.
3. **Clean ARIA.** `aria-controls` on the handle is just the DOM id of the
   previous Pane sibling.
4. **Configuration on Root.** Per-pane constraints stay in one map.

### Single constraint per side

The boundary is a single value (0–100), so the constraint surface is minimal:
one `minSize` per pane. A pane's upper bound is the complement of its partner's
`minSize` (`max = 100 − partner.minSize`), which the clamp and the handle's
`aria-valuemax` both derive. There is intentionally no per-pane `maxSize` —
it would be a second source of truth for the same boundary edge.

### Float precision

Sizes carry full float precision end-to-end (`defaultSizes` → clamp → drag →
`onSizesChange(End)` → `width: …%`). Nothing in the size pipeline rounds. The
only rounding is the handle's `aria-valuenow` (and `aria-valuetext`), which is a
screen-reader announcement and does not affect layout. This lets consumers hit
an exact pixel at a known width (`31.25` = 250px in an 800px container).

### 2-pane primitive, composable via nesting

`Splitter.Root` accepts exactly two `Pane` children with one `Handle` between
them. For 3+ regions, nest one Splitter inside the children of a `Pane`:

```tsx
<Splitter.Root defaultSizes={{ nav: 25, rest: 75 }}>
  <Splitter.Pane id="nav">…</Splitter.Pane>
  <Splitter.Handle />
  <Splitter.Pane id="rest">
    <Splitter.Root defaultSizes={{ main: 65, aside: 35 }}>
      <Splitter.Pane id="main">…</Splitter.Pane>
      <Splitter.Handle />
      <Splitter.Pane id="aside">…</Splitter.Pane>
    </Splitter.Root>
  </Splitter.Pane>
</Splitter.Root>
```

The 2-pane shape is what the W3C window splitter ARIA pattern is specified
against. Nesting gives screen-reader users a tree of self-contained widgets,
each matching the W3C example exactly.

## Persistence

Persistence lives in the consuming app, not in the component — hydrate
`defaultSizes` from any storage and write back on `onSizesChangeEnd`. Collapse
is controlled state, so it persists the same way.

```tsx
// `useLocalStorage` is any storage hook of your choosing (cookies, query
// params, server state, sessionStorage, …).
const [sizes, setSizes] = useLocalStorage("ide", { nav: 30, main: 70 });
const [collapsed, setCollapsed] = useLocalStorage("ide-collapsed", null);

<Splitter.Root
  defaultSizes={sizes}
  onSizesChangeEnd={setSizes}
  collapsedPane={collapsed}
  onCollapsedPaneChange={setCollapsed}
  panes={{ nav: { minSize: 10, collapsible: true }, main: { minSize: 20 } }}
>
  …
</Splitter.Root>;
```

`defaultSizes` is read once on mount and normalized to sum 100, so a slightly
drifted stored value self-heals. Unknown ids in storage have no effect (sizes
are looked up by the current pane ids).

## Disabled state

`isDisabled` on Root makes the whole splitter non-interactive: the handle is
removed from the tab order (`tabIndex={-1}`), gets `aria-disabled="true"`, and
ignores drag, keyboard, and collapse input.

```tsx
<Splitter.Root isDisabled defaultSizes={{ nav: 30, main: 70 }}>
  …
</Splitter.Root>
```

## Out of scope

Intentionally not supported on this primitive:

- **3+ panes per splitter** — use nesting.
- **Cascading resize.** With only two panes there is nowhere to spill
  remainder; drag clamps at the boundary.
- **Controlled `sizes` prop.** Sizes stay uncontrolled for drag performance;
  persist via `onSizesChangeEnd` + any storage.
- **Imperative command API.** Collapse is controllable state; there is no ref
  or hook to wire.
- **Per-pane `maxSize`.** Derived from the partner's `minSize`.
- **Pixel / rem / vh units.** Percentages only (full float precision). True
  cross-resize pixel pinning is a possible later addition.
- **App-shell behaviour** (responsive collapse to drawers, landmark slots).
  That belongs to a separate pattern component (`AppLayout`) which composes
  `Splitter`.
- **Per-handle configuration.** A 2-pane splitter has one handle; behaviour
  is configured on `Root`.
