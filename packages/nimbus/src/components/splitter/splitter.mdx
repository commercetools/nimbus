# Splitter

A 2-pane compound primitive for user-resizable layouts. The handle between the
two `Splitter.Pane`s is draggable (mouse, touch, and keyboard), supports
collapse / expand, and meets the W3C window splitter ARIA pattern.

## Overview

`Splitter.Root` accepts exactly two `Splitter.Pane` children with one
`Splitter.Handle` between them. Each pane carries a string `id`; all per-pane
configuration (sizes, constraints, collapsibility) lives on `Root` keyed by
that id. The component is **uncontrolled** — for cross-component imperative
commands and persistence, use the companion `useSplitterLayout` hook.

For three or more regions, **nest** one Splitter inside another's `Pane`.

### Resources

- [W3C ARIA APG window splitter pattern](https://www.w3.org/WAI/ARIA/apg/patterns/windowsplitter/)

## Guidelines

### Best practices

- **Two panes per Splitter.** Nest splitters for layouts with 3+ regions; each
  nested splitter is independently focusable, persistable, and announced to
  assistive technology.
- **Set per-pane `minSize`.** Without a floor, drag can shrink a pane below
  its usable content size. Pair `minSize` with content that scrolls inside the
  pane (e.g. wrap content in `ScrollArea`).
- **Use `collapsible: true` for chrome panes.** A nav or aside that the user
  may want hidden should be `collapsible` so double-click and Enter on the
  handle toggle it.
- **Persist with `useSplitterLayout`.** Give consumers a stable layout across
  sessions; the hook handles SSR, debouncing, and gracefully reconciles
  stored data that drifts from the current pane set.

### Usage

The Splitter is the primitive for user-driven resize: IDE shells, document /
preview side-by-side, file-tree + editor, master / detail.

> [!TIP]\
> When to use
- **Side-by-side content** the user benefits from resizing themselves.
- **App shells** that compose a fixed-position nav with a flexible main area.
- **Master/detail** where the relative split is part of the user's workflow.

> [!CAUTION]\
> When not to use
- **Static two-column layouts** — use a flex or grid layout directly.
- **Mobile-first, narrow viewports** — drag interactions are awkward; use a
  `Drawer` or stack the regions vertically.
- **More than two peers per splitter** — nest splitters instead of trying to
  jam three panes onto one Root.

## Basic usage

A horizontal splitter with sensible per-pane defaults.

```jsx-live
const App = () => (
  <Box height="400px" width="100%" borderWidth="25" borderColor="neutral.6">
    <Splitter.Root
      orientation="horizontal"
      panes={{
        nav: { defaultSize: 30, minSize: 15 },
        main: { defaultSize: 70, minSize: 25 },
      }}
    >
      <Splitter.Pane id="nav">
        <ScrollArea h="100%" w="100%">
          <Box p="400" bg="indigo.3" minH="100%">
            <Heading size="md" mb="200">Nav</Heading>
            <Text>Drag the handle on the right to resize.</Text>
          </Box>
        </ScrollArea>
      </Splitter.Pane>
      <Splitter.Handle />
      <Splitter.Pane id="main">
        <ScrollArea h="100%" w="100%">
          <Box p="400" bg="amber.3" minH="100%">
            <Heading size="md" mb="200">Main</Heading>
            <Text>Try keyboard navigation: Tab to focus the handle, then ArrowLeft / ArrowRight.</Text>
          </Box>
        </ScrollArea>
      </Splitter.Pane>
    </Splitter.Root>
  </Box>
);
```

## Orientation

`orientation="vertical"` stacks the panes top-to-bottom; ArrowUp / ArrowDown
become the active keys, and `aria-orientation="vertical"` is announced on the
handle (per W3C separator semantics — orientation describes the boundary
axis, not the layout axis).

```jsx-live
const App = () => (
  <Box height="500px" width="100%" borderWidth="25" borderColor="neutral.6">
    <Splitter.Root
      orientation="vertical"
      panes={{
        top: { defaultSize: 40 },
        bottom: { defaultSize: 60 },
      }}
    >
      <Splitter.Pane id="top">
        <Box p="400" bg="teal.3" h="100%">
          <Heading size="md" mb="200">Top</Heading>
        </Box>
      </Splitter.Pane>
      <Splitter.Handle />
      <Splitter.Pane id="bottom">
        <Box p="400" bg="rose.3" h="100%">
          <Heading size="md" mb="200">Bottom</Heading>
        </Box>
      </Splitter.Pane>
    </Splitter.Root>
  </Box>
);
```

## Per-pane constraints

`minSize` and `maxSize` clamp the boundary on drag and keyboard. They also
drive the handle's `aria-valuemin` / `aria-valuemax` announcements.

```jsx-live
const App = () => (
  <Box height="400px" width="100%" borderWidth="25" borderColor="neutral.6">
    <Splitter.Root
      panes={{
        nav: { defaultSize: 30, minSize: 15, maxSize: 50 },
        main: { defaultSize: 70, minSize: 25 },
      }}
    >
      <Splitter.Pane id="nav">
        <Box p="400" bg="indigo.3" h="100%">
          <Heading size="md">Nav (min 15 / max 50)</Heading>
        </Box>
      </Splitter.Pane>
      <Splitter.Handle />
      <Splitter.Pane id="main">
        <Box p="400" bg="amber.3" h="100%">
          <Heading size="md">Main (min 25)</Heading>
        </Box>
      </Splitter.Pane>
    </Splitter.Root>
  </Box>
);
```

## Collapsible panes

A pane configured with `collapsible: true` toggles between its `defaultSize`
and `collapsedSize` (default 0) via double-click on the handle or Enter when
the handle is focused. `onCollapse(paneId)` / `onExpand(paneId)` fire on each
transition. Use `disableDoubleClick` on Root to disable double-click while
keeping the keyboard shortcut.

```jsx-live
const App = () => (
  <Box height="400px" width="100%" borderWidth="25" borderColor="neutral.6">
    <Splitter.Root
      panes={{
        nav:  { defaultSize: 30, minSize: 10, collapsible: true },
        main: { defaultSize: 70, minSize: 20 },
      }}
      onCollapse={(id) => console.log("collapsed:", id)}
      onExpand={(id) => console.log("expanded:", id)}
    >
      <Splitter.Pane id="nav">
        <Box p="400" bg="indigo.3" h="100%">
          <Heading size="md">Nav (double-click handle to collapse)</Heading>
        </Box>
      </Splitter.Pane>
      <Splitter.Handle />
      <Splitter.Pane id="main">
        <Box p="400" bg="amber.3" h="100%">
          <Heading size="md">Main</Heading>
        </Box>
      </Splitter.Pane>
    </Splitter.Root>
  </Box>
);
```

## Persistence and cross-app commands

The Splitter is **uncontrolled** by design: drag fires at ~60Hz, so threading
state through a controlled prop re-renders the consumer tree on every tick.
The companion `useSplitterLayout` hook gives you the same outcomes as a
controlled component (persistence, imperative collapse / expand from anywhere)
without the re-render cost.

```tsx
import { Splitter, useSplitterLayout } from "@commercetools/nimbus";

const layout = useSplitterLayout({
  initialSizes: { nav: 30, main: 70 },
  id: "ide-layout", // → localStorage key, with SSR-safe sync hydration
});

<Splitter.Root
  panes={{
    nav:  { minSize: 10, collapsible: true },
    main: { minSize: 20 },
  }}
  defaultSizes={layout.defaultSizes}
  onSizesChange={layout.onSizesChange}
  __layoutRef={layout.__layoutRef}
>
  <Splitter.Pane id="nav">…</Splitter.Pane>
  <Splitter.Handle />
  <Splitter.Pane id="main">…</Splitter.Pane>
</Splitter.Root>

// Elsewhere in the app, with access to `layout`:
layout.collapse("nav");
layout.expand("nav");
layout.setSizes({ nav: 0, main: 100 });
```

For non-localStorage backends, pass a custom `storage` adapter (cookies,
sessionStorage, server state):

```ts
useSplitterLayout({
  initialSizes: { nav: 30, main: 70 },
  storage: {
    load: () => readFromCookie(),
    save: (sizes) => writeToCookie(sizes),
  },
});
```

## Three or more regions: nesting

The Splitter is intentionally 2-pane. For 3+ regions, nest a Splitter inside
the children of a `Pane`. Each splitter has its own state and can be
persisted, collapsed, and announced to AT independently.

```jsx-live
const App = () => (
  <Box height="500px" width="100%" borderWidth="25" borderColor="neutral.6">
    <Splitter.Root panes={{ nav: { defaultSize: 25 }, rest: { defaultSize: 75 } }}>
      <Splitter.Pane id="nav">
        <Box p="400" bg="indigo.3" h="100%">
          <Heading size="md">Nav</Heading>
        </Box>
      </Splitter.Pane>
      <Splitter.Handle />
      <Splitter.Pane id="rest">
        <Splitter.Root panes={{ main: { defaultSize: 65 }, aside: { defaultSize: 35 } }}>
          <Splitter.Pane id="main">
            <Box p="400" bg="amber.3" h="100%">
              <Heading size="md">Main</Heading>
            </Box>
          </Splitter.Pane>
          <Splitter.Handle />
          <Splitter.Pane id="aside">
            <Box p="400" bg="rose.3" h="100%">
              <Heading size="md">Aside</Heading>
            </Box>
          </Splitter.Pane>
        </Splitter.Root>
      </Splitter.Pane>
    </Splitter.Root>
  </Box>
);
```

## Keyboard reference

When `Splitter.Handle` has focus:

| Key             | Behaviour                                                            |
| --------------- | -------------------------------------------------------------------- |
| ArrowLeft / ArrowRight (horizontal) | Move the boundary by `keyboardStep` percentage points. |
| ArrowUp / ArrowDown (vertical)      | Same, on the vertical axis.                            |
| Home            | Shrink the previous pane to its `minSize`.                           |
| End             | Grow the previous pane to its `maxSize` (or `100 − next.minSize`).   |
| Enter           | Toggle collapse of the adjacent collapsible pane.                    |
