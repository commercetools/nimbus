## Overview

The Steps component helps users understand where they are in a multi-step
process. It provides clear visual feedback about completed, current, and
upcoming steps, making complex workflows feel more manageable and predictable.

### Key Features

- **Three size variants**: Extra small (xs), small (sm), and medium (md) to fit
  different interface densities
- **Two orientations**: Horizontal for wide layouts and vertical for sidebar or
  narrow contexts
- **Flexible indicators**: Numeric step numbers with optional checkmarks, or
  custom icons for semantic meaning
- **Built-in navigation**: Includes navigation triggers and content panels with
  automatic state management

## Guidelines

Use the Steps component strategically to enhance user workflow.

### Best practices

- **Keep labels concise**: Use short, clear step labels that describe the action
  or content
- **Limit step count**: Avoid more than 5-7 steps to prevent overwhelming users
- **Provide descriptions**: Use descriptions for complex steps that need
  additional context
- **Maintain consistency**: Use the same indicator pattern throughout a flow

> [!TIP]\
> When to use
>
> - **Multi-step forms**: Registration, checkout, surveys
> - **Wizards**: Setup flows, configuration, onboarding
> - **Progress tracking**: Task completion, status indicators

> [!CAUTION]\
> When not to use
>
> - **Linear navigation**: Use tabs or pagination instead
> - **Single actions**: Steps are meant for multi-step processes
> - **Unknown step count**: Use progress bars for indeterminate progress

## Accessibility

Accessibility ensures that digital content and functionality are usable by
everyone, including people with disabilities, by addressing visual, auditory,
cognitive, and physical limitations.

### Accessibility standards

- **Semantic structure**: Uses tablist semantics (`role="tablist"` and
  `role="tab"`) for proper screen reader navigation
- **Current step indication**: Uses `aria-selected="true"` to announce the
  active step
- **Panel association**: Triggers reference their content panels via
  `aria-controls`
- **Orientation**: Communicated via `aria-orientation` attribute
- **Keyboard navigation**: Tab moves focus between triggers; Enter/Click
  activates a trigger
- **Color independence**: State is communicated through both color AND visual
  indicators (checkmarks, styling) to support users with color vision deficiency
