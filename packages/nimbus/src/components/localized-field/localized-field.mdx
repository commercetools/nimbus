# Localized field

A localized field is a single component that enables users to input and manage content for multiple language versions of a piece of data within one unified form area.

## Overview

The localized field component serves as a container that displays and manages content across various defined locales. It minimizes clutter by allowing users to toggle between language versions using a control while ensuring that the data for each specified locale is correctly captured, validated, and stored in the application.

## Variables

Get familiar with the features.



### Size

There are two supported sizes for localized fields, the default (md) and a smaller version for more compact needs (sm).


```jsx-live
const App = () => {
  const [smValues, setSmValues] = useState({ en: 'Hello world', de: 'Hallo Welt' });
  const [mdValues, setMdValues] = useState({ en: 'Hello world', de: 'Hallo Welt' });

  return (
    <Box>
      <LocalizedField
        size="md"
        defaultLocaleOrCurrency="en"
        valuesByLocaleOrCurrency={mdValues}
        onChange={(e) => {
          setMdValues({
            ...mdValues,
            [e.target.locale]: e.target.value
          });
        }}
      />
       <LocalizedField
        size="sm"
        defaultLocaleOrCurrency="en"
        valuesByLocaleOrCurrency={smValues}
        onChange={(e) => {
          setSmValues({
            ...smValues,
            [e.target.locale]: e.target.value
          });
        }}
      />
    </Box>
  );
};
```

### Expand and hide

Allow users to be able to expand and minimize localized fields to simplify a page, or expand to see everything when needed.


```jsx-live
const App = () => {
  const [values, setValues] = useState({ en: 'Hello world', de: 'Hallo Welt', es: 'Hola Mundo', nl: 'Hallo Wereld' });

  return (
    <LocalizedField
      size="md"
      defaultLocaleOrCurrency="en"
      defaultExpanded
      valuesByLocaleOrCurrency={values}
      onChange={(e) => {
        setValues({
          ...values,
          [e.target.locale]: e.target.value
        });
      }}
    />
  );
};
```


### Localized field combinations

Combine with rich text, text, multiline text, richttext, and moneyinput to give users more availability for localized input details.


```jsx-live
const App = () => {
  const [enValues, setEnValues] = useState({
    en: 'hello',
  });

  const [deValues, setDeValues] = useState({
    de: 'hallo',
  });

  return (
    <Box>
      <LocalizedField
        type="richText"
        defaultLocaleOrCurrency="en"
        valuesByLocaleOrCurrency={enValues}
        onChange={(e) => {
          setEnValues({
            ...enValues,
            [e.target.locale]: e.target.value
          });
        }}
      />
      <LocalizedField
        type="richText"
        defaultLocaleOrCurrency="de"
        valuesByLocaleOrCurrency={deValues}
        onChange={(e) => {
          setDeValues({
            ...enValues,
            [e.target.locale]: e.target.value
          });
        }}
      />
    </Box>
  );
};
```

## Guidelines

Localized field guidelines ensure the consistent design, accessibility, and efficient management of content that requires simultaneous input and storage across multiple defined language versions.

### Best practices

- **Prioritize default locale:** Always ensure the primary or default locale is easily accessible and the first field a user sees.
- **Use clear flags/labels:** The language switcher controls must clearly and concisely indicate the active locale to prevent accidental input into the wrong language version.
- **Maintain content consistency:** The underlying input component chosen for renderItem must remain the same for every single locale instance to ensure data integrity and predictable behavior.
- **Handle text expansion:** Design the surrounding layout and container with enough horizontal space to accommodate text expansion in wordier languages like German or French.
- **Provide contextual tools:** If the underlying input is complex (like a rich text editor), ensure localization features (like spellcheck based on the active locale) are supported.

### Usage

A localized field's usage is to efficiently capture and manage content that must be translated and stored for multiple language versions within a single form.

> [!TIP]\
> When to use

- **Product descriptions/titles:** For any field that must be translated for customer-facing display (e.g., product titles, category names, SEO keywords).
- **Multiple language input:** When a user (typically a content manager or merchant) is responsible for entering content for all supported markets from one interface.
- **Structured metadata:** For input fields that hold localized metadata (e.g., unique localized pricing tiers, currency symbols, SEO, or tax explanations).

> [!CAUTION]\
> When not to use

- **Non-translatable data:** When capturing internal IDs, SKUs, inventory counts, or purely numerical data that does not change by language. Use a standard Text Input or Number Input instead.
- **Single-language input:** If the application supports only one language, or if the user is only ever responsible for one market's content.
- **Large, multi-page content:** For entire pages or long-form documentation that would require a complex translation workflow; use a dedicated content management system instead.


### Do’s and don’ts for the localized field



> [!TIP]\
> **Do**
>
> - Ensure switching languages preserves the content of the input field currently being edited, allowing easy comparison.
> - If needed, you can include a visual indicator with a country or language code on the active input field for clarity.


> [!CAUTION]\
> **Don't**
>
> - Don't hide the language switcher interface completely unless an external system (like a global header) explicitly manages it.
> - Don't allow the underlying input component to change across locales (e.g., using a text input for English and a text area for French).
> - Don't allow users to input data into a locale that is not defined in the component's locales variable.


## Accessibility

Accessibility ensures that digital content and functionality are usable by everyone, including people with disabilities, by addressing visual, auditory, cognitive, and physical limitations.

```jsx-live

  const [values, setValues] = useState({ en: 'Hello world', de: 'Hallo Welt' });
```

### Accessibility standards

 - **Focus management:** Ensure the Tab order moves logically between the active input field and the language switching controls. When the language is switched, focus should return to the newly visible input field.
 - **Keyboard operability:** The language switching controls (tabs, buttons) must be fully navigable and operable via the keyboard.
 - **ARIA roles:** Ensure the active input field correctly inherits all necessary ARIA attributes (e.g., aria-required, aria-invalid) from the FormField component.
 - **Screen reader communication:** Ensure screen readers announce the change of language context when the user activates a different locale input, often by announcing the new tab or language code.
