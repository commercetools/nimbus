# Button Group

---

A set of closely related, mutually exclusive or complementary actions that are important enough to be displayed directly in the interface for quick access.

## Overview

Button groups can be used several ways, with and without icons, and spanning from two selections to several.

Deep dive on details and access design library.

### Visual options

Toggle Button Groups can be configured with labels, icons, or both.

#### Labels and icons

Button groups with icons and labels are used to present a set of related actions in a compact and visually clear way, where icons provide quick recognition and labels offer explicit understanding of each option. This combination enhances user efficiency by making common choices readily available and easily identifiable.

```jsx-live
const App = () => (
  <ToggleButtonGroup.Root aria-label="Labels and icons example" selectedKeys={["li-shipped"]} tone="primary">
    <ToggleButtonGroup.Button id="li-processing">
      <Icons.RotateRight /> Processing
    </ToggleButtonGroup.Button>
    <ToggleButtonGroup.Button id="li-shipped">
      <Icons.LocalShipping /> Shipped
    </ToggleButtonGroup.Button>
    <ToggleButtonGroup.Button id="li-cancelled">
      <Icons.Dangerous /> Cancelled
    </ToggleButtonGroup.Button>
    <ToggleButtonGroup.Button id="li-refunded">
      <Icons.CurrencyExchange /> Refunded
    </ToggleButtonGroup.Button>
    <TooltipTrigger delay={0} closeDelay={0}>
      <ToggleButtonGroup.Button id="li-more-options" aria-label="More Options">
        <Icons.MoreVert />
      </ToggleButtonGroup.Button>
      <Tooltip>More Options</Tooltip>
    </TooltipTrigger>
  </ToggleButtonGroup.Root>
);
```

#### Just labels

Button groups with just labels provide explicit understanding of each option. This approach prioritizes readability and ease of comprehension, making it suitable for scenarios where icons might be ambiguous or when the actions are best described through text.

## Guidelines

Button group guidelines focus on presenting sets of related actions in a clear, organized, and accessible manner, enabling users to easily understand and choose between the available options, often emphasizing visual connection and logical arrangement.

### Best practices

-   **Order logically:** Arrange buttons by importance or user flow.
-   **Use clear labels:** Keep button text concise and understandable, if using icons, utilize tooltip labels.
-   **Limit button count:** Avoid overwhelming users with too many options.
-   **Space appropriately:** Place button groups close to controlled content.
-   **Ensure accessibility:** Make them navigable and understandable by everyone.

### Usage

A button group is best used when you need to present users with a set of closely related, mutually exclusive (e.g., a user selecting data table views), or complementary actions that are important enough to be displayed directly in the interface for quick access. Think of scenarios where users need to make a choice between a few options or perform a series of connected tasks within a specific context.

> [!TIP]\
> When to use
-   **Grouping related actions:** Organize similar tasks together. Use button groups to cluster actions that are logically connected, like order processing steps or product editing options.
-   **Providing mutually exclusive choices:** Offer clear alternatives. When users need to select only one option from a set, like filtering by order status or choosing a report timeframe.
-   **Streamlining workflows:** Enable quick task completion. For frequently performed actions on a single item or within a specific context, like managing inventory or editing customer details.
-   **Conserving screen space:** Present options compactly. When screen real estate is limited, button groups offer a space-efficient way to display multiple relevant actions.
-   **Enhancing visual clarity:** Improve comprehension of options. Visually connecting related buttons helps users understand the available choices and their relationships.
-   **Facilitating efficient navigation:** Offer quick jumps within a section. For navigating between closely related views or steps within a specific management area.

> [!CAUTION]\
> When not to use
-   **Do not use a button group as a tab navigation:** There is a specific [tab component](#), please use this component instead. There is better accessibility support, and consistency with other designs and uses.
-   **Unrelated actions:** Don't group unrelated actions together. If buttons perform completely different tasks, a button group can be confusing.
-   **Overcrowding the UI:** Avoid too many buttons within a group. Too much choice confuses users and makes decision making more difficult. Tailor actions for the user by presenting common actions upfront and nesting other actions under a "more" icon or where they can be featured better.
-   **Do not use for multi-stepped processes:** Button groups are for single click actions, not for guiding users through complex sequences.
-   **Avoid if actions aren't logically linked:** If there is no clear relationship between the buttons, grouping them may be misleading.

### Selecting views

To allow users to switch between different display views (e.g., grid view, list view, map view). Make sure that users can easily recognize icons and pair with hover tooltips.

> [!TIP]\
> **Do**
>
> -   Do use clear differentiated, and familiar icons for users to be able to toggle options.
> -   Do use tooltips to easily explain what a tool is to a user.

```jsx-live
const App = () => (
  <ToggleButtonGroup.Root aria-label="Select view" selectedKeys={["view-list"]} tone="primary">
    <TooltipTrigger delay={0} closeDelay={0}>
      <ToggleButtonGroup.Button id="view-grid" aria-label="Grid View">
        <Icons.Apps />
      </ToggleButtonGroup.Button>
      <Tooltip>Grid View</Tooltip>
    </TooltipTrigger>
    <TooltipTrigger delay={0} closeDelay={0}>
      <ToggleButtonGroup.Button id="view-list" aria-label="List View">
        <Icons.List />
      </ToggleButtonGroup.Button>
      <Tooltip>List View</Tooltip>
    </TooltipTrigger>
    <TooltipTrigger delay={0} closeDelay={0}>
      <ToggleButtonGroup.Button id="view-map" aria-label="Map View">
        <Icons.ViewWeek />
      </ToggleButtonGroup.Button>
      <Tooltip>Week View</Tooltip>
    </TooltipTrigger>
  </ToggleButtonGroup.Root>
);
```

> [!CAUTION]\
> **Don't**
>
> -   Do not use icons that users will not easily recognize or pairings of icons that seem too visually similar.
> -   Always pair icon buttons with tooltip labels for all users.

const App = () => (

### Filter information

In data tables or lists, button groups can be used to filter or sort content.

> [!TIP]\
> **Do**
>
> -   When using button groups to show filtered lists, pair closely with information to make sure that users know that this interaction shows tagged information instead of changing the state of any selected data. This can sometimes look like an item count included with the button group items.
> -   You may amend a "more" button at the end that is icon only to organize more options if needed. Otherwise it is not good practice to mix styles in button groupings.

```jsx-live
const App = () => (
  <ToggleButtonGroup.Root aria-label="Filter status" selectedKeys={["filter-accepted"]} tone="primary">
    <ToggleButtonGroup.Button id="filter-drafts">
      Drafts (6)
    </ToggleButtonGroup.Button>
    <ToggleButtonGroup.Button id="filter-accepted">
      Accepted (2)
    </ToggleButtonGroup.Button>
    <ToggleButtonGroup.Button id="filter-rejected">
      Rejected (10)
    </ToggleButtonGroup.Button>
  </ToggleButtonGroup.Root>
);
```

> [!CAUTION]\
> **Don't**
>
> -   Do not confuse the user with unclear options.
> -   Do not use unfamiliar and lengthly labels.

## Selection Modes

By default, the ToggleButtonGroup allows only one button to be selected at a
time, and allows deselection by clicking the currently selected button. You can
control this behavior with the `selectionMode` prop:

```jsx-live
const App = () => (
  <Stack direction="column" gap="400">
    <Text>Single Selection with Deselection Allowed (default)</Text>
    <ToggleButtonGroup.Root
      aria-label="Default selection behavior"
      selectionMode="single"
    >
      <ToggleButtonGroup.Button id="single-left">Left</ToggleButtonGroup.Button>
      <ToggleButtonGroup.Button id="single-center">Center</ToggleButtonGroup.Button>
      <ToggleButtonGroup.Button id="single-right">Right</ToggleButtonGroup.Button>
    </ToggleButtonGroup.Root>

    <Text>Multiple Selection</Text>
    <ToggleButtonGroup.Root
      aria-label="Multiple selection example"
      selectionMode="multiple"
    >
      <ToggleButtonGroup.Button id="multiple-bold">Bold</ToggleButtonGroup.Button>
      <ToggleButtonGroup.Button id="multiple-italic">Italic</ToggleButtonGroup.Button>
      <ToggleButtonGroup.Button id="multiple-underline">Underline</ToggleButtonGroup.Button>
    </ToggleButtonGroup.Root>
  </Stack>
);
```

## Accessibility

The ToggleButtonGroup component is built with accessibility in mind:

- Implements proper ARIA roles (uses `radiogroup` and `radio` roles)
- Supports keyboard navigation using arrow keys
- Provides appropriate focus management
- Requires an `aria-label` for the group to ensure screen reader users
  understand its purpose

### Accessibility Standards

#### 1. Perceivable:

-   **1.1.1 Non-text Content (Level A):**
    -   Provide text alternatives for any non-text content (e.g., icons within buttons) via `aria-label` on the button or surrounding `Tooltip`.
    -   Ensure icons have sufficient visual clarity.
-   **1.4.1 Use of Color (Level A):**
    -   Avoid relying solely on color to indicate the selected state. Default styles typically use background color and potentially borders or font weight changes.
-   **1.4.3 Contrast (Minimum) (Level AA):**
    -   Ensure sufficient color contrast between button text/icons and their background, in both selected and unselected states.
    -   Ensure sufficient contrast between adjacent buttons.

#### 2. Operable:

-   **2.1.1 Keyboard (Level A):**
    -   Make the entire group focusable using the Tab key.
    -   Allow navigation between buttons within the group using arrow keys (Left/Right or Up/Down depending on orientation).
    -   Allow selection using the Enter or Space key.
-   **2.4.4 Link Purpose (In Context) (Level A):** (Applies conceptually to button purpose)
    -   Button labels (text or `aria-label` for icons) should clearly describe the action or choice.
-   **2.4.7 Focus Visible (Level AA):**
    -   Provide a clear and visible focus indicator for the currently focused button within the group.

#### 3. Understandable:

*   **3.2.1 On Focus (Level A):**
    -   Ensure that moving focus between buttons does not trigger unexpected actions.
*   **3.3.2 Labels or Instructions (Level A):**
    -   Provide a clear `aria-label` for the `ToggleButtonGroup.Root` element to describe the purpose of the group to assistive technologies.

#### 4. Robust:

*   **4.1.2 Name, Role, Value (Level A):**
    -   Use appropriate ARIA roles (e.g., `radiogroup` for single selection, `group` for multiple selection, `radio` or `checkbox` for items depending on mode) to ensure compatibility with assistive technologies. Nimbus components built on React Aria generally handle this automatically.
