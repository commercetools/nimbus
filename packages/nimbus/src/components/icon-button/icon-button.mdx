# Icon Button

Provides a compact and visually efficient way to represent actions, especially
when space is limited or the action is universally understood.

## Overview

These specifications define the standard styles and variations for icon buttons
within our design system, covering different visual treatments for diverse use
cases.

### Resources

Deep dive on details and access design library.

[Figma library](https://www.figma.com/design/AvtPX6g7OGGCRvNlatGOIY/NIMBUS-design-system?node-id=346-2570&m=dev)

## Hierarchy

#### Most common: Ghost

Ghost icon buttons are used most often, the represent familiar actions that
don't compete with a primary action on the page.

```jsx-live
const App = () => {
  return (
    <IconButton variant="ghost" tone="primary">
      <Icons.Edit/>
    </IconButton>
  );
};
```

#### Use sparingly: Solid

Solid icon buttons are sparingly used, but represent commonly understood primary
actions by the user.

```jsx-live
const App = () => {
  return (
    <IconButton variant="solid" tone="primary">
      <Icons.Edit/>
    </IconButton>
  );
};
```

#### Least used: Outline

Outline icon buttons are used as secondary buttons, used often in button groups
to represent icon toggles and within data tables.

```jsx-live
const App = () => {
  return (
    <IconButton variant="outline" tone="primary">
      <Icons.Edit/>
    </IconButton>
  );
};
```

## Variables

Get familiar with the features.

### Size

#### Medium (40px)

These are the default and preferred size for icon buttons.

```jsx-live
const App = () => {
  return (
    <IconButton variant="solid" tone="primary" size="md">
      <Icons.Edit/>
    </IconButton>
  );
};
```

#### X-Small (32px)

Placed in a small dense content areas, e.g., in pop-over or inside a data table.

```jsx-live
const App = () => {
  return (
    <IconButton variant="solid" tone="primary" size="xs">
      <Icons.Edit/>
    </IconButton>
  );
};
```

#### XX-Small (24px)

2XS is used sparingly mostly inside other components e.g, inside inputs or
filter badges.

```jsx-live
const App = () => {
  return (
    <IconButton variant="solid" tone="primary" size="2xs">
      <Icons.Edit/>
    </IconButton>
  );
};
```

### Tone

#### Primary

This is the default tone for the icon button. This styling should be used most
commonly for all actions.

```jsx-live
const App = () => {
  return (
    <IconButton variant="ghost" tone="primary">
      <Icons.ViewColumn/>
    </IconButton>
  );
};
```

#### Critical

Use this icon button when the user is approaching a permanent destructive
action. See guidelines for more information.

```jsx-live
const App = () => {
  return (
    <IconButton variant="ghost" tone="critical">
      <Icons.Delete/>
    </IconButton>
  );
};
```

#### Tooltip

For accessibility compliance (and good user experience for all) all icon buttons
must have a tooltip. Go to tooltips for more formation about the component.

```jsx-live
const App = () => {
  return (
    <TooltipTrigger delay={0} closeDelay={0}>
      <IconButton variant="ghost" tone="primary">
        <Icons.Edit/>
      </IconButton>
      <Tooltip>
        Edit
      </Tooltip>
    </TooltipTrigger>
  );
};
```

## Guidelines

Icon buttons are interactive elements that empower users to trigger actions and
navigate within your product. They serve as clear calls to action, guiding users
through key flows and enabling them to interact with content in meaningful ways.
These guidelines ensure consistent styling and behavior for all buttons,
contributing to a cohesive and predictable user experience.

### Writing guidelines

- Use
  [Material Icons](https://mui.com/material-ui/material-icons/?theme=Outlined&query=access#search-material-icons)
  in outline style before adding a new custom icon.
- Use standard icons.
- Provide tooltips.
- Combine with text when needed.
- Be consistent.
- Test with users.

### Usage

#### Common & effective placements:

- **Toolbars**: Icon buttons are perfect for toolbars due to their compact size.
  They allow for many actions to be readily available without cluttering the
  interface.
- **Navigation** bars/headers: Often used for actions like search, settings, or
  menu access. Placement here makes these key functions easily accessible
  throughout the app.
- **Within** lists/grids: Icon buttons can be placed next to list items or grid
  elements to provide actions specific to that item (e.g., edit, delete, share).
- **Contextual** menus: Icon buttons can be used within dropdown menus or
  context menus to provide a visual representation of available actions.

#### Placement considerations:

- **Related to content**: Place icon buttons close to the content they affect.
  For example, an edit button should be near the text it's used to edit.
- **Consistency**: Maintain consistent placement of similar icon buttons. Users
  expect to find common actions in the same place.
- **Groupings**: If you have multiple related icon buttons, grouping them
  together visually. Helps users understand the relationships between the
  actions.
- **Avoid clutter**: Don't overcrowd your UI with too many icon buttons.

> [!TIP]\
> When to use

- Space is limited.
- Actions are universally recognized.
- Reinforcing meaning.
- Adding visual appeal.
- Supporting multiple languages.
- Interfaces are simple and consistent.

> [!CAUTION]\
> When not to use

- Actions are ambiguous.
- Context is crucial.
- Actions are critical.
- Space is plentiful.
- Interfaces are complex.
- Testing reveals confusion.

### Tone and style usage

Icons that are on the left are the best placement. Icons are used to lead the
user to the right action quickly by scanning. Ok ways to break this rule are
with drop downs and right pointing arrows.

#### Most common: Ghost

They are often seen next to tables, fields, and toolbars.

```jsx-live
const App = () => {
  return (
    <Stack direction="horizontal" justifyContent="center">
      <IconButton tone="primary" variant="ghost">
        <Icons.ViewColumn />
      </IconButton>
      <IconButton tone="primary" variant="ghost">
        <Icons.Clear />
      </IconButton>
    </Stack>
  )
}
```

#### Solid and Outline

Solid icon buttons are most commonly seen in navigation. Outline buttons are
most commonly seen in groups of two representing views that can be toggled.

```jsx-live
const App = () => {
  return (
    <Stack direction="horizontal" justifyContent="center">
      <IconButton tone="primary" variant="solid">
        <Icons.Edit />
      </IconButton>
      <IconButton tone="primary" variant="outline">
        <Icons.Add />
      </IconButton>
    </Stack>
  )
}
```

#### Critically toned

Reserved for destructive actions, such as deleting something or removing a
connection.

```jsx-live
const App = () => {
  return (
    <Stack direction="horizontal" justifyContent="center">
      <IconButton tone="critical" variant="outline">
        <Icons.Delete />
      </IconButton>
      <IconButton tone="critical" variant="solid">
        <Icons.Delete />
      </IconButton>
      <IconButton tone="critical" variant="ghost">
        <Icons.Delete />
      </IconButton>
    </Stack>
  )
}
```

### Button groupings

Button groups are UI elements that organize multiple buttons together, often
visually as a single unit. They serve to present users with a set of related
choices or actions in a compact and organized way.

> [!TIP]\
> **Do**
>
> - If you use a stack of icon buttons for actions, use max 4 and hide the rest
>   within a "more" icon.

```jsx-live
const App = () => {
  return (
    <Stack direction="horizontal" gap="0">
      <IconButton tone="primary" variant="ghost">
        <Icons.Edit />
      </IconButton>
      <IconButton tone="primary" variant="ghost">
        <Icons.PushPin />
      </IconButton>
      <IconButton tone="primary" variant="ghost">
        <Icons.Delete />
      </IconButton>
      <IconButton tone="primary" variant="ghost">
        <Icons.MoreVert />
      </IconButton>
    </Stack>
  )
}
```

> [!CAUTION]\
> **Don't**
>
> - Avoid having all the options as solid primary filled buttons.
> - Try not to have icon buttons sandwiched between buttons.

```jsx-live
const App = () => {
  return (
    <Stack direction="horizontal">
      <IconButton tone="primary" variant="solid">
        <Icons.MoreVert />
      </IconButton>
      <Button tone="primary" variant="solid">
        Save changes
      </Button>
      <IconButton tone="primary" variant="solid">
        <Icons.ManageSearch />
      </IconButton>
      <Button tone="primary" variant="solid">
        <Icons.Undo />
        Revert
      </Button>
```

> **Do**
>
> - When offering a toggle experience, connect these icon buttons

### Tooltips

Make sure to place tooltips in areas where they will be seen easily.

> [!TIP]\
> **Do**
>
> - Place tooltips so that it makes information easy to read.

```jsx-live
  return (
    <Stack width="185px">
      <TooltipTrigger delay={0} closeDelay={0} isOpen>

        </IconButton>

          Table settings

        aria-label="Select a category"
        selectedKey=""
      >
        <Select.Options>
          <Select.Option id="productColumns" value="productColumns">
```

## Accessibility

The component uses `onPress*` props to handle all user interactions—mouse
clicks, touch taps, and keyboard presses—consistently. These props (`onPress`,
`onPressStart`, `onPressEnd`, etc.) ensure a smooth and accessible experience
regardless of the user's input method or device.


```jsx-live
const App = () => {
  return (
    <IconButton tone="primary" variant="solid" onPress={() => alert('Button pressed!')}>
      <Icons.Edit />
    </IconButton>
  )
}
```

### Accessibility Standards

#### 1. Perceivable:

- **Sufficient Color Contrast:** Ensure enough contrast between the icon and its
  background. WCAG requires a contrast ratio of at least 3:1 for graphical
  elements and 4.5:1 for text. Tools like WebAIM's color contrast checker can
  help you verify this.
- **Non-text Content:** Icons are considered non-text content. WCAG requires
  that all non-text content has a text alternative that serves the equivalent
  purpose. This usually means providing an alternative text (alt text) for the
  icon.
- **Visual Cues:** Don't rely solely on color to convey information. Use
  additional visual cues like shapes, patterns, or text labels to ensure that
  the icon's meaning is clear to everyone, including people with color
  blindness.

#### 2. Operable:

- **Keyboard Accessibility:** Icon buttons must be operable using a keyboard.
  Users should be able to navigate to the button using the Tab key and activate
  it using the Enter or Space key.
- **Target Size:** Ensure that the icon button is large enough to be easily
  clickable, especially on touchscreens. WCAG recommends a minimum target size
  of 24x24 CSS pixels.
- **Focus Indicator:** Provide a clear visual indicator when the icon button
  receives keyboard focus. This helps users understand which element is
  currently active.

#### 3. Understandable:

- **Clear Labels:** If the icon's meaning isn't universally understood, provide
  a text label or tooltip to clarify its purpose. This is especially important
  for less common icons.
- **Consistent Design:** Use icons consistently throughout your website. This
  helps users learn the meaning of each icon and makes it easier for them to
  navigate your site.

#### 4. Robust:

- **Assistive Technology Compatibility:** Ensure that icon buttons are
  compatible with assistive technologies like screen readers. This often
  involves using proper HTML markup and ARIA attributes to provide semantic
  information about the button.
