# Search Index Truncation: Problem and Options

## Problem

The `search_docs` MCP tool searches a pre-built index (`search-index.json`)
generated by `nimbus-docs-build`. At build time, each doc page's MDX content is
stripped of markdown formatting and **truncated to 500 characters** before being
written to the index.

For component pages, the first 500 characters typically contain only the
overview blurb and resource links. Prop names, API details, behavioral
documentation, and usage examples are deeper in the page and invisible to
search.

### Impact

- 74% of index entries (105 of 142) hit the 500-char cap
- Only 6 of 77 component pages mention "prop" within the first 500 chars
- Queries for specific terms like `onChange`, `disabled`, or `variant` return
  few or zero matches despite being extensively documented

The search still finds the **correct page** via title, description, and tags
(e.g. "button" finds the Button page), but fails when searching by **specific
content** like prop names or behavioral details.

### Only the overview tab is indexed

Each component page has multiple views stored in the route files:

| View         | Button example | Contains                              |
| ------------ | -------------- | ------------------------------------- |
| `overview`   | 4,765 chars    | Description, hierarchy, visual specs  |
| `dev`        | 12,017 chars   | Import paths, props, usage examples   |
| `guidelines` | 15,339 chars   | Design guidelines, do/don't patterns  |
| `a11y`       | 2,700 chars    | Keyboard interactions, ARIA details   |

The search index generator reads `doc.mdx`, which equals `views.overview.mdx`.
The `dev`, `guidelines`, and `a11y` views are **not indexed at all**. For the
Button component, this means 87% of the content (30K of 35K chars) is invisible
to search — and that's exactly where props, imports, and accessibility details
live.

### Where the truncation happens

`packages/nimbus-docs-build/src/generators/search.ts`, line 43:

```ts
content: stripMarkdown(doc.mdx).slice(0, 500),
```

## Options

### Option 1: Increase the content limit

Raise the truncation from 500 to 2000+ characters.

| Limit | Entries truncated | Index size growth |
| ----- | ----------------- | ----------------- |
| 500   | 74%               | baseline          |
| 2000  | 50%               | ~4x content       |
| 5000  | 27%               | ~10x content      |

**Pros:**

- Simplest change (one number)
- Captures more content from every page

**Cons:**

- 50% of pages still truncated at 2000 chars (median page is 1920 chars)
- Diminishing returns: the longest pages (often the most important component
  docs) need 5000–10000+ chars to cover fully
- Linearly increases index size and memory usage for Fuse.js

### Option 2: Extract section headings and prop names into tags

At index build time, parse section headings (`## Props`, `## Keyboard
Interactions`, etc.) and prop/API names from the full MDX and append them to the
existing `tags` array. The content field stays at 500 chars.

**Pros:**

- Targets exactly what's missing (prop names, section headings, API terms)
- Minimal index size growth (tags are short deduplicated strings)
- Works regardless of page length
- Tags are already weighted at 2x in the Fuse.js config

**Cons:**

- Requires parsing logic in `nimbus-docs-build` to extract headings/prop names
- Extracted terms are unranked (no positional context)
- Needs maintenance as MDX structure evolves

### Option 3: Add a dedicated `keywords` field with extracted terms

Similar to option 2 but uses a separate `keywords` field instead of overloading
`tags`. Pull unique terms from the full MDX — prop names, code tokens, heading
text — into a deduplicated keyword list. Add a corresponding Fuse.js key with
appropriate weight.

**Pros:**

- Keeps authored `tags` separate from auto-extracted terms
- Can tune Fuse.js weight independently from tags
- More structured and explicit

**Cons:**

- Requires a schema change to the search index (new field)
- Needs updates in both `nimbus-docs-build` (generation) and `nimbus-mcp`
  (search config)
- Slightly more implementation effort than option 2

### Option 4: Two-phase search with lazy-loaded route files (recommended)

Use the lightweight search index for candidate discovery (phase 1), then
lazy-load the full route JSON files for matched candidates and search across all
views (phase 2). No build-time changes required — the route files already
contain complete MDX for all views (`overview`, `dev`, `guidelines`, `a11y`).

**How it works:**

```
Query: "onChange"

Phase 1 — Lightweight index (title, description, tags)
  → Candidates: Select, DatePicker, Combobox, TextField, ... (by tags/title)
  → Fast: ~5ms, searches 120 KB in-memory index

Phase 2 — Lazy-load route files for top ~15 candidates
  → Read full route JSONs from disk (~300 KB total)
  → Search all views (overview + dev + guidelines + a11y) for query
  → Re-rank: pages where query appears in content get boosted
  → Return top results with snippets from actual match locations
  → Latency: ~10–30ms (SSD reads + string search)

Fallback: if phase 1 returns <5 candidates, expand to all
component-tagged pages to avoid missing deep-content-only matches.
```

**Performance comparison:**

| Metric              | Current    | Opt 1 (2000) | Opt 2 (tags) | Opt 4 (lazy) |
| ------------------- | ---------- | ------------ | ------------ | ------------ |
| Memory at rest      | 120 KB     | ~470 KB      | ~130 KB      | ~120 KB      |
| Disk I/O per query  | 0          | 0            | 0            | ~150–450 KB  |
| Latency per query   | <5ms       | <5ms         | <5ms         | ~10–30ms     |
| Content coverage    | 500 chars  | 2000 chars   | Headings     | **All views** |
| Finds `onChange`?   | No         | Maybe        | If extracted | **Yes**      |
| Finds a11y details? | No         | No           | If extracted | **Yes**      |
| Build-time changes  | None       | 1 line       | Parser needed | **None**    |
| Views searched      | Overview   | Overview     | Overview     | **All 4**   |

**Pros:**

- 100% content coverage — every word in every view is searchable
- Zero build-time changes — route files already exist with all views
- Memory stays flat — only the lightweight index is held in memory
- Snippets come from actual match locations, not truncated previews
- Naturally extends to new views/tabs without maintenance

**Cons:**

- ~10–30ms latency per query (negligible for MCP tool calls where LLM round
  trips take seconds)
- Disk I/O on every query (mitigated by OS file cache after first call)
- Fallback path (scanning all component routes) needed for queries that don't
  match any metadata
- More implementation complexity than options 1–3

## Recommendation

Option 4 (two-phase lazy-load) is the recommended approach. It solves the root
problem completely — all four views become searchable without any build pipeline
changes. The latency cost (~10–30ms) is invisible in the context of MCP tool
calls. Options 1–3 are partial fixes that either leave content gaps or require
build-time extraction logic that needs ongoing maintenance.

The route files with full view content already exist on disk. This approach
simply reads them at query time instead of trying to cram everything into an
index at build time.
